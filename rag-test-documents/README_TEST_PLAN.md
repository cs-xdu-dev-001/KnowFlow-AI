# KnowFlow AI RAG 测试方案

## 1. 上传顺序

第一轮只上传 01、02、03、07 四个 KnowFlow 相关文档，测试基础问答和引用片段。

第二轮上传 04、05、06，测试跨项目总结、表格解析、DOCX 解析和简历改写能力。

第三轮上传 08、09，测试冲突资料处理。新版设计应优先于旧版草稿。

第四轮上传 10，测试无关资料干扰。系统不应把无关文档当作依据。

## 2. 推荐问题

- 总结 KnowFlow AI 的核心功能和技术亮点。
- 为什么知识库创建时要绑定 Embedding 模型？
- 当前最新版 KnowFlow 是否支持 GitHub OAuth 和 Gemini？
- 把校园行为异常识别微调项目改写成简历项目经历。
- 本地生活探店平台中 Redis、JWT 和 ThreadLocal 分别解决什么问题？
- 这个知识库里有没有说明如何养猫？

## 3. 通过标准

- 回答应显示引用片段，且引用来源与问题相关。
- 冲突文档场景下，应区分旧版草稿和新版设计。
- 无关问题应说明资料不足，不应编造。
- DOCX、PPTX、XLSX、JSON、CSV、MD、TXT 均应能解析并展示入库状态。
- 前端文档状态应能看到上传中、解析中、切分完成、向量化完成、失败原因和重试入口。

## 4. 重点观察

- 引用片段是否准确。
- 回答是否把旧设计当成当前设计。
- 表格和 PPTX 中的信息是否能被检索出来。
- 多轮追问是否沿用当前 session 的上下文。
- 不选择知识库时是否仍然能正常普通对话。
